# Руководство по настройке маршрута 3D-модели HERO13

В этом руководстве объясняется, где настраиваются положение, маршрут,
вращение и адаптивный размер модели HERO13, как объединяются параметры и
какой профиль используется при каждой ширине экрана.

## Краткая карта файлов

| Задача | Файл | Основные настройки |
| --- | --- | --- |
| Точки маршрута и преобразования профилей | `src/js/three-route.js` | `BASE_ROUTE`, `PROFILE_SETTINGS` |
| Брейкпоинты и адаптивный размер | `src/js/three-responsive.js` | `getRouteProfile()`, `getResponsiveModelScale()` |
| Применение позиции, вращения и масштаба | `src/js/three-scene.js` | `applyRouteTransform()` |
| Тайминг прокрутки и видимость маршрута | `src/js/scroll-scenes.js` | `refreshProductRoute()`, экземпляры ScrollTrigger |
| Видимая область WebGL | `src/styles.css` | `.webgl-stage` |

## Активный профиль в зависимости от ширины

Функция `getRouteProfile(width)` выбирает профиль маршрута:

| Ширина viewport | Активный профиль |
| --- | --- |
| Меньше `768px` | `compactPhone` |
| `768px`–`1023px` | `compactTablet` |
| `1024px`–`1199px` | `tablet` |
| `1200px` и больше | `desktop` |

Важно: существующий профиль `mobile` сейчас не выбирается.
`getRouteProfile()` возвращает `compactPhone` раньше, чем
`getViewportProfile()` может вернуть `mobile`. Поэтому изменение `mobile`
при текущей логике не даёт видимого результата.

Чтобы снова активировать его, замените:

```js
if (width < 768) return "compactPhone";
```

на:

```js
if (width < 768) return "mobile";
```

Это вернёт боковое движение, но также может снова вызвать пересечение модели
с текстом на узких экранах.

## Порядок точек маршрута

`BASE_ROUTE` содержит четыре точки прокрутки:

```text
Индекс 0: Hero
Индекс 1: Performance
Индекс 2: Stability
Индекс 3: Rugged
```

Все массивы `scales` и `rotations` внутри `PROFILE_SETTINGS` используют
именно этот порядок.

WebGL-маршрут заканчивается на Rugged. Когда Battery достигает отметки `72%`
viewport, WebGL-область плавно скрывается и рендеринг останавливается.
Battery, Mounting, Lenses, Comparison и Preorder не являются точками
маршрута.

## Параметры точки маршрута

Каждая точка `BASE_ROUTE` имеет следующую структуру:

```js
{
  id: "hero",
  selector: "#top",
  anchorX: 0.62,
  anchorY: 0.06,
  z: 0,
  rotation: { x: -0.08, y: -0.42, z: -0.06 },
  scale: 1,
}
```

### `anchorX`: положение по горизонтали

```text
Отрицательное значение → влево
0                      → центр
Положительное значение → вправо
```

Практический начальный диапазон — примерно от `-0.6` до `0.6`. Значения,
близкие к `-1` или `1`, приближают модель к границе камеры и могут обрезать
её.

### `anchorY`: положение по вертикали

```text
Отрицательное значение → вниз
0                      → центр
Положительное значение → вверх
```

Для настройки также удобно начинать с диапазона от `-0.6` до `0.6`.

### `z`: глубина

```text
Положительное значение → ближе к камере
Отрицательное значение → дальше от камеры
```

Изменение `z` также влияет на видимый размер из-за перспективы. Для
предсказуемого управления размером используйте `scale`, а `z` оставьте для
управления глубиной.

### `rotation`

Значения вращения задаются в радианах:

| Значение | Примерный угол |
| --- | --- |
| `0.1` | `5.7°` |
| `0.5` | `28.6°` |
| `1` | `57.3°` |

Оси:

```text
x → наклон вперёд/назад
y → поворот влево/вправо
z → боковой наклон
```

### `scale`

`1` — базовый масштаб маршрута, `1.1` — увеличение на 10%, `0.9` —
уменьшение на 10%.

В текущей реализации каждый профиль содержит собственный массив `scales`.
Функция `getProductRoute()` перезаписывает `BASE_ROUTE.scale` значением
`profile.scales[index]`. Поэтому изменение только `BASE_ROUTE.scale` ничего
не изменит, пока существует переопределение в профиле.

## Как настройки профиля преобразуют базовый маршрут

Для каждой точки `getProductRoute()` рассчитывает:

```js
finalAnchorX = baseAnchorX * profile.xAmplitude;

finalAnchorY =
  baseAnchorY * (profile.yAmplitude ?? 1) +
  profile.yOffset;

finalRotation =
  profile.rotations?.[index] ?? baseRotation;

finalRouteScale = profile.scales[index];
```

### `xAmplitude`

Управляет шириной всего горизонтального маршрута:

| Значение | Результат |
| --- | --- |
| `0` | Все точки находятся по центру |
| `0.3` | Небольшое боковое движение |
| `0.7` | Заметное движение |
| `1` | Полный маршрут из `BASE_ROUTE` |
| Больше `1` | Маршрут шире базового |

Это множитель, а не общий горизонтальный сдвиг.

### `yAmplitude`

Определяет, насколько сохраняются базовые вертикальные различия между
точками:

| Значение | Результат |
| --- | --- |
| `0` | Все различия `anchorY` удаляются |
| `0.5` | Вертикальные различия уменьшаются вдвое |
| `1` или параметр отсутствует | Сохраняются полные различия |

### `yOffset`

Перемещает весь маршрут по вертикали:

```text
Более отрицательное значение → весь маршрут ниже
Значение ближе к 0           → ближе к центру WebGL-области
Положительное значение       → весь маршрут выше
```

Аналогичного параметра `xOffset` сейчас нет. Для точного горизонтального
позиционирования нужно изменять `BASE_ROUTE.anchorX`, `xAmplitude` или
расширить код способом, описанным далее.

## Текущее положение на телефоне

Активный телефонный профиль:

```js
compactPhone: {
  xAmplitude: 0,
  yAmplitude: -1,
  yOffset: -0.38,
  scales: [1.6, 1.7, 2, 1.8],
  // ...
}
```

Его итоговые координаты:

```js
finalAnchorX = anyBaseX * 0;
// Результат: 0 для каждой точки

finalAnchorY = baseAnchorY * -1 + -0.38;
// Результаты: Hero -0.44, Performance -0.38,
// Stability -0.42, Rugged -0.36
```

Модель остаётся по центру горизонтали, но следует по небольшому вертикальному
маршруту. Масштабы точек равны `1.6`, `1.7`, `2` и `1.8`.

Начальное состояние страницы — первая точка активного профиля. Поэтому
дефолтное положение всегда соответствует точке Hero выбранного профиля.

Активный профиль compact-tablet остаётся в точке `x: 0`, `y: 0` и использует
масштабы маршрута `[1.6, 1.8, 1.7, 1.7]`.

## Перемещение всего телефонного маршрута

Чтобы поднять весь маршрут, сохранив его вертикальные различия:

```js
compactPhone: {
  xAmplitude: 0,
  yAmplitude: -1,
  yOffset: -0.2,
  scales: [1.6, 1.7, 2, 1.8],
  // ...
}
```

По сравнению с текущим `-0.38`, значение `-0.2` ближе к центру и поэтому
располагает модель выше.

Сравнение:

```text
-0.50 → переместить весь маршрут ниже
-0.38 → текущее смещение
-0.20 → переместить весь маршрут выше
 0     → центрировать маршрут вокруг инвертированной базы
```

Чтобы зафиксировать одну вертикальную позицию во всех точках, установите
`yAmplitude: 0`; тогда будет использоваться только `yOffset`.

## Возвращение движения на телефоне

Пример:

```js
compactPhone: {
  xAmplitude: 0.7,
  yAmplitude: 1,
  yOffset: -0.1,
  scales: [1, 1.06, 0.98, 1.03],
  // ...
}
```

Итоговые горизонтальные координаты:

| Точка | Расчёт | Результат |
| --- | --- | --- |
| Hero | `0.62 × 0.7` | `0.434` |
| Performance | `0.42 × 0.7` | `0.294` |
| Stability | `-0.54 × 0.7` | `-0.378` |
| Rugged | `0.40 × 0.7` | `0.280` |

Маршрут:

```text
Справа → немного справа → слева → справа
```

## Изменение отдельной точки

Например, можно изменить базовую точку Stability:

```js
{
  id: "stability",
  selector: "#stability",
  anchorX: -0.3,
  anchorY: 0.15,
  z: -0.1,
  rotation: { x: -0.18, y: 0.46, z: -0.14 },
  scale: 1.04,
}
```

Это переместит Stability ближе к горизонтальному центру и выше, сохранив
модель немного дальше от камеры.

Изменение `BASE_ROUTE` влияет на все профили с ненулевой амплитудой. Это не
является независимым переопределением только для телефона.

## Изменение размера модели в каждой секции

Изменяйте массив `scales` активного профиля:

```js
compactPhone: {
  scales: [
    1.2,  // Hero
    1.1,  // Performance
    1.05, // Stability
    1.15, // Rugged
  ],
}
```

Результат:

```text
Hero        → 120% базового масштаба маршрута
Performance → 110%
Stability   → 105%
Rugged      → 115%
```

## Адаптивный размер

Масштаб маршрута — только одна часть итогового размера:

```text
Итоговый размер модели =
адаптивный масштаб viewport × масштаб точки маршрута
```

Адаптивный масштаб рассчитывается функцией `getResponsiveModelScale()` в
`src/js/three-responsive.js`.

Основные константы:

```js
const MIN_PHONE_SCALE = 0.52;
const MIN_PHONE_SCALE_COMPACT = 0.64;
const MIN_LANDSCAPE_SCALE = 0.58;
const FULL_SIZE_WIDTH = 1200;
const FULL_SIZE_HEIGHT = 720;
const FULL_SIZE_STAGE_ASPECT = 1.6;
```

Функция рассчитывает значения по ширине, высоте и соотношению сторон
WebGL-области, после чего выбирает минимальное:

```js
return Math.min(widthScale, heightScale, aspectScale);
```

Пример:

```text
Адаптивный масштаб: 0.64
Масштаб Hero:       1.20
Итоговый масштаб:   0.64 × 1.20 = 0.768
```

Чтобы увеличить минимальный размер модели при ширине меньше `1024px`:

```js
const MIN_PHONE_SCALE_COMPACT = 0.75;
```

Это минимальный, а не фиксированный размер. Итоговое значение по-прежнему
может ограничиваться шириной, высотой или соотношением сторон WebGL-области.

## WebGL-область и центр координат

На desktop `.webgl-stage` занимает весь viewport:

```css
.webgl-stage {
  position: fixed;
  inset: 0;
}
```

При ширине меньше `1024px` CSS изменяет область:

```css
.webgl-stage {
  top: 29vh;
  height: 58vh;
}
```

Поэтому `anchorY: 0` означает центр полосы высотой `58vh`, а не центр всей
страницы. Если координаты математически выглядят правильными, но модель
визуально расположена слишком высоко или низко, сначала проверьте CSS
WebGL-области и только потом увеличивайте смещения маршрута.

## Интерполяция маршрута и тайминг прокрутки

Модель не перескакивает между точками. Позиция, глубина, вращение и масштаб
интерполируются через `smoothstep`:

```js
progress = progress * progress * (3 - 2 * progress);
```

Scroll anchors рассчитываются по реальной геометрии секций:

```js
sectionTop +
sectionHeight / 2 -
viewportHeight * 0.35
```

Изменение `anchorX`, `anchorY`, `z`, вращения или масштаба изменяет
преобразование модели, но не момент достижения точки. Тайминг прокрутки
управляется отдельно этой формулой и диапазоном ScrollTrigger.

## Рекомендуемый стартовый маршрут для `compactPhone`

```js
compactPhone: {
  xAmplitude: 0.45,
  yAmplitude: 1,
  yOffset: -0.15,

  scales: [
    1.15, // Hero
    1.1,  // Performance
    1.05, // Stability
    1.12, // Rugged
  ],

  rotations: [
    { x: -0.05, y: -0.3, z: -0.03 },
    { x: 0, y: -0.15, z: 0 },
    { x: -0.05, y: 0.15, z: -0.03 },
    { x: 0.05, y: -0.3, z: 0.02 },
  ],
}
```

Этот вариант возвращает заметное движение, но оставляет его более узким и
спокойным, чем на desktop.

## Независимые позиции для каждого профиля

Текущий формат профиля поддерживает:

- одну горизонтальную амплитуду для всего маршрута;
- одну вертикальную амплитуду для всего маршрута;
- одно вертикальное смещение для всего маршрута;
- независимые вращение и масштаб для каждой точки.

Он не поддерживает точные переопределения `anchorX`/`anchorY` для каждой
точки только одного профиля.

Если нужны точные координаты только для телефона, расширьте профиль:

```js
compactPhone: {
  positions: [
    { x: 0, y: -0.2 },    // Hero
    { x: 0.2, y: -0.1 },  // Performance
    { x: -0.2, y: 0 },    // Stability
    { x: 0.1, y: -0.15 }, // Rugged
  ],
  // ...
}
```

Затем измените `getProductRoute()`:

```js
const position = profile.positions?.[index];

return {
  // ...
  anchorX: position?.x ?? node.anchorX * profile.xAmplitude,
  anchorY:
    position?.y ??
    node.anchorY * (profile.yAmplitude ?? 1) +
      profile.yOffset,
  // ...
};
```

После этого каждый профиль сможет содержать собственные точные координаты
без изменения маршрутов desktop и tablet.

## Проверка значений во время работы

Посмотреть текущие значения можно в Browser Console:

```js
document.querySelector(".webgl-stage").dataset
```

Важные поля:

```text
modelScale → адаптивный масштаб viewport
routeX     → текущая целевая горизонтальная координата
routeY     → текущая целевая вертикальная координата
```

Вывести только эти значения:

```js
const stage = document.querySelector(".webgl-stage");

console.table({
  responsiveScale: stage.dataset.modelScale,
  routeX: stage.dataset.routeX,
  routeY: stage.dataset.routeY,
});
```

`modelScale` не включает масштаб текущей точки маршрута. Для получения
полного масштаба умножьте его на соответствующее значение из массива
`scales` активного профиля.

## Безопасный порядок настройки

1. Проверьте ширину viewport в Browser Console.
2. По таблице ширины определите активный профиль.
3. Изменяйте только одну категорию за раз:
   - положение;
   - амплитуду маршрута;
   - масштаб;
   - вращение.
4. Проверьте все четыре секции при прокрутке в обе стороны.
5. Проверьте обрезание модели и пересечение с текстом.
6. Проверьте точные границы брейкпоинтов: `767`, `768`, `1023`, `1024`,
   `1199` и `1200` пикселей.
7. Запустите:

```bash
npm test
npm run build
```

При изменении маршрутов или профилей обычно требуется обновить ожидаемые
значения в:

```text
tests/three-route.test.js
tests/three-responsive.test.js
```
